01 პრობლემა — რა გვჭირდება

Sazeo IoT/BMS პლატფორმაზე რამდენიმე სცენარი არსებობს ერთდროულად:

🏢 გარედან რეგისტრაცია
ნებისმიერ კომპანიას შეუძლია self-register — შექმნას org და დაიწყოს პლატფორმის გამოყენება invite-ის გარეშე.
🔗 Cross-org წვდომა
კომპანია A-ს შეუძლია მოიწვიოს კომპანია B, რომ B-ს user-ები A-ს resource-ებს შეზღუდულად ნახავდნენ.
🛠 სერვის-პროვაიდერი
კონტრაქტორი (მაგ. HVAC კომპანია) შეიძლება "ჩემოდეს" კლიენტთან სერვისის მიმწოდებლად — შეზღუდული, revocable წვდომით.
👁 DMT / Platform Operator
DMT-ი ხედავს ყველა org-ს system-admin დონეზე. ეს არ არის cross-tenant grant — ეს platform-level privilege.

02 ორგანიზაციის ტიპები

სამი ტიპი — DB enum org_type:

🏛 Platform Operator platform_operator

  • DMT (მხოლოდ ერთი)
  • ხედავს ყველა org-ს
  • Grant-ის გარეშე — system privilege
  • user-ები: system_admin role

🏢 Client Company client

  • self-register ან platform-invite
  • ხედავს მხოლოდ საკუთარ data-ს
  • შეუძლია სხვა org-ს მოიწვიოს
  • მათ user-ებს: org_admin, member, viewer

🛠 Service Provider service_provider

  • კონტრაქტორი / მომსახურე კომპანია
  • საკუთარი data — ისეთივე isolated
  • კლიენტ org-ებში — მხოლოდ granted scope-ით
  • კლიენტი revoke-ავს ნებისმიერ დროს
ℹ
org_type-ი არ განსაზღვრავს permission-ებს — org_membership.role და org_access_grant.scope განსაზღვრავს. ტიპი მხოლოდ UI/UX logics-ს და billing plan-ს ეხება.

03 Self-registration flow

გარე კომპანიას შეუძლია დარეგისტრირდეს — ისე, რომ DMT-ს წინასწარი invite არ სჭირდება:

1
Email + password რეგისტრაცია Firebase Auth → UID. POST /auth/register → user ჩაიწერება users table-ში status: pending_verification.
2
Email verification Firebase Send email verification. მანამ — API-ი limited access-ს ბრუნებს. user-ს შეუძლია login, მაგრამ org-ი ჯერ არ გაქვს.
3
კომპანიის პროფილის შექმნა POST /organizations — user ქმნის org-ს: სახელი, org_type, ქვეყანა. ავტომატურად ეძლევა org_membership(role: org_admin).
4
Platform verification (optional gate) თუ requires_platform_approval = true — org status pending_approval. DMT system_admin ადასტურებს. Manual gate production-ში spam/fraud-ის საწინააღმდეგოდ.
5
Active — სრული წვდომა org.status = active. user-ს შეუძლია devices, members, billing და სხვა.
Registration state machine
  // org.status enum
  pending_verification   → email-ი ჯერ არ არის confirmed
         ↓
  pending_approval       → email OK, DMT-ი ადასტურებს (თუ gate ჩართულია)
         ↓
  active                 → სრული წვდომა
         ↓
  suspended             → billing overdue / policy violation
  terminated            → soft-deleted, data retained 90d (GDPR)
⚠
Anti-spam gate: rate-limit POST /organizations — max 3 orgs per IP per 24h. Email domain blocking (disposable emails). Production-ში requires_platform_approval = true პირველ 6 თვეში.

04 Invite-code cross-org access (GitHub model)

GitHub-ის organization invite-ის ანალოგი — Org A-მ შეუძლია Org B-ს კოდი გაუგზავნოს, B-მ კოდი შეიყვანოს, granted access-ი გაჩნდეს.

Invite flow — ორ მხარის პერსპექტივა
  ─── Org A (grantor) ───────────────────────────────────────

  POST /organizations/:orgId/invites
  {
    scope: "read:devices,read:reports",   // scope enum-ით
    expiresInDays: 30,
    label: "TBC Facilities team"           // UI-ში ნახვისთვის
  }

  ← {
    inviteCode: "SAZEO-A3X9-2026",         // humanreadable, 16-char
    expiresAt:  "2026-05-23T00:00:00Z",
    status:     "pending"
  }

  Org A admin გადასცემს კოდს B-ს (email / copy-paste / Slack)


  ─── Org B (grantee) ───────────────────────────────────────

  POST /invites/accept
  {
    inviteCode: "SAZEO-A3X9-2026",
    orgId:      "org_b_uuid"             // B-ს org — user-ი B-ს member უნდა იყოს
  }

  ← {
    grantId:  "grant_xyz",
    grantorOrg: { name: "სავაგო ენერგეტიკა", id: "org_a" },
    scope:    "read:devices,read:reports",
    status:   "active"
  }

  ─── შემდეგ session-ში ─────────────────────────────────────

  B-ს user-ი Header-ში აგზავნის:
  Organization: org_a_uuid           // A-ს context-ში ოპერირებს
  X-Grant-Id: grant_xyz              // optional — logging-ისთვის

  Guard ამოწმებს: B-ს org_access_grants-ში არსებობს A-ზე active grant?
  თუ კი — scope-ის ფარგლებში წვდომა. თუ არა — 404.

Invite code format

SAZEO-XXXX-YYYY — prefix + 4-char segment + 4-char segment

DB-ში ინახება SHA-256 hash (plaintext — არასოდეს). Single-use: accept-ის შემდეგ status → accepted, კოდი invalid ხდება.

Brute-force protection: POST /invites/accept — max 10 attempts / IP / hour. Wrong code → generic error (not "code doesn't exist" vs "code expired" — enumeration prevention).

Scope enum

  enum InviteScope {
    read_devices     = "read:devices",      // მოწყობილობების ნახვა
    read_reports     = "read:reports",      // ანგარიშების ნახვა
    read_alerts      = "read:alerts",       // სიგნალების ნახვა
    write_commands   = "write:commands",    // ბრძანებების გაგზავნა
    full_access      = "full",              // სრული წვდომა (org_admin ახდენს grant-ს)
  }

05 Service Provider Pattern

სცენარი: კლიენტ კომპანია იყენებს გარე კონტრაქტორ კომპანიას (HVAC, ელექტრო, security). კონტრაქტორს სჭირდება კლიენტის IoT სისტემაში წვდომა, მაგრამ მხოლოდ შეზღუდული, revocable, auditable.

ეს ისეთივე invite-code flow-ია — განსხვავება ტიპშია

1
სერვის-პროვაიდერი self-registers ქმნის საკუთარ org-ს org_type: service_provider.
2
კლიენტი invite-ს გენერირებს scope: "read:devices,write:commands" + label "TBC HVAC contractor".
3
სერვის-პროვაიდერი accept-ავს ახლა მათ user-ებს შეუძლიათ კლიენტის context-ში ოპერირება — scope-ის ფარგლებში.
4
კლიენტი revoke-ავს DELETE /organizations/:orgId/grants/:grantId — instant. სერვის-პროვაიდერი წყვეტს წვდომას.

Audit trail — ყველა cross-tenant action-ი logged

  // user_audit_log table entry
  {
    actor_user_id:   "user_sp_john",
    actor_org_id:    "org_sp",       // სპ-ს org
    target_org_id:   "org_client",   // კლიენტის org
    grant_id:        "grant_xyz",    // via რომელი grant
    action:          "device.read",
    resource_id:     "device_abc",
    timestamp:       "2026-04-23T10:22:00Z",
    ip:              "185.x.x.x"
  }
✓
კლიენტს UI-ში ეჩვენება: "ბოლო 30 დღეში სერვის-პროვაიდერმა ეს actions შეასრულა" — სრული visibility.

07 Security Invariants

🔒
Cross-tenant access-ი = explicit grant ან system_admin. სხვა შემთხვევაში — 404 (არა 403). 403-ი ადასტურებს resource-ის არსებობას. 404 — neutral.
INV-001Guard logic — 3 validation layerRequired
  1. JWT valid + not expired — standard Firebase JWT check
  2. Organization header normalization — normalizeOrgHeader() rejects array, trims whitespace (ADR-009)
  3. Membership OR grant check — user-ი ან target org-ის member-ია, ან active grant-ი აქვს target org-ზე. Platform operator — პირდაპირ passes.
INV-002Scope validation — not just org membershipRequired
Cross-tenant context-ში guard ამოწმებს grant.scope-ს. თუ endpoint-ს @RequireScope('write:commands') აქვს, მაგრამ grant-ს მხოლოდ read:devices — 403.
INV-003Invite token — single-use + hashed + time-limitedRequired
token_hash = SHA-256(plaintext_code). Plaintext-ი DB-ში არ ინახება. Accept-ზე status → accepted → კოდი invalid. Expired invite-ის accept → 404 (generic).
INV-004Meta-test: cross-tenant IDOR preventionDay-1
Jest meta-test: org_a-ს user-ი org_b-ს resource-ებს ითხოვს (without grant) → assert 404. ყოველი tenant-scoped entity-სთვის. მემკვიდრეობა: dmt-web-app test_tenant_isolation.py 9 regression test.

08 ADR Decisions

MT-001 Invite code — humanreadable, not UUID Locked
Decision: SAZEO-XXXX-YYYY format (prefix + 8 random alphanum chars).
Rationale: UUID-ები (e.g. f47ac10b-58cc-4372-a567) გადაცემისა და შეყვანისათვის შეცდომისადმია მოწყვლადი. Humanreadable format-ი (GitHub-style) გაცილებით user-friendly. Security — SHA-256 hash DB-ში უზრუნველყოფს.
Alternatives considered: UUID → rejected (UX). QR code → additional feature layer, not primary.
MT-002 Grant scope — string list, not bitmask Locked
Decision: Comma-separated scope strings "read:devices,read:reports", not integer bitmask.
Rationale: Human-readable in DB, easy to debug, extensible without migration. Bitmask — compresses well but makes debugging, audit log reading, and scope UI complexity worse. At Sazeo scale (<50 scope types), string list გაცილებით readable-ია.
MT-003 UNIQUE(grantor, grantee) — ერთი active grant per pair Locked
Decision: org_access_grants-ში UNIQUE constraint (grantor_org_id, grantee_org_id).
Rationale: Multiple grants between same org pair creates scope confusion and revoke complexity. თუ scope-ი გასაფართოვებელია — grant-ი update-ავს, არ ემატება ახალი row. Simpler mental model for both devs and ops.
MT-004 Self-registration — open by default, platform-gate available Open — configure per-env
Decision: organizations.requires_approval per-org flag, plus global env REQUIRE_ORG_APPROVAL=true for production.
Rationale: Dev/staging — open signup for rapid testing. Production — gated to prevent spam and fraudulent orgs. Flag allows gradual rollout (open beta → gated).
