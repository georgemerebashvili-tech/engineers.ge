02Guard Decision Flow — 5 Layer

  Request შემოდის
         │
         ▼
  ┌─────────────────────────────────────────────────────────────────────┐
  │ Layer 1 — Rate Limit (Redis)                                        │
  │   global IP limit  →  429   org quota limit  →  429                   │
  └─────────────────────────────────────────────────────────────────────┘

         ▼

  │                                                                     │

  │         │ no                                                        │
  │   JWT present?   no  ──────────────────────────────── → 401        │
  │         │ yes                                                       │
  │   Firebase verify JWT                                               │
  │         │ invalid  ──────────────────────────────────── → 401       │
  │         │ valid → user_id extracted                                 │
  │   user.status = active?  no  ──────────────────────── → 403        │
  └─────────────────────────────────────────────────────────────────────┘
         │ OK — user identified
         ▼
  ┌─────────────────────────────────────────────────────────────────────┐
  │ Layer 3 — Organization Context                                      │
  │                                                                     │
  │   @NoOrg route (profile, select-org)?  ─────────────── → PROCEED   │
  │         │ no                                                        │
  │   Organization header present?  no  ───────────────── → 400        │
  │         │ yes → normalizeOrgHeader()                               │
  │   Array / malformed?  ──────────────────────────────── → 400        │

  │   user is member of org?  ──── no ──────────────────── → 404        │
  │   OR active cross-tenant grant?                                     │
  │         │ yes → orgContext injected                                │
  └─────────────────────────────────────────────────────────────────────┘
         │ OK — org context valid
         ▼
  ┌─────────────────────────────────────────────────────────────────────┐
  │ Layer 4 — Role / Scope                                              │
  │                                                                     │
  │   @RequireScope() declared?  no  ──────────────────── CI FAIL       │
  │         │ yes — scope list checked                                  │
  │   Direct member?   →  check org_membership.role ≥ required         │
  │   Cross-tenant?    →  check org_access_grant.scope contains req    │
  │   System admin?    →  always pass                                   │
  │   role/scope insufficient  ────────────────────────── → 403        │
  └─────────────────────────────────────────────────────────────────────┘
         │ OK
         ▼
  ┌─────────────────────────────────────────────────────────────────────┐
  │ Layer 5 — Resource Tenancy                                          │
  │                                                                     │
  │   every DB query:  .where({ id, organizationId: ctx.orgId })       │
  │   result empty?  ───────────────────────────────────── → 404        │

  │   → PROCEED to controller                                           │

ℹ

05Routing Rules — 12 წესი

1Public route = @Public() decorator — explicit whitelist
Next.js middleware და NestJS guard ყველა route-ს auth-required-ად მიიჩნევს default-ად. Public-ი მხოლოდ @Public() annotation-ით გახდება. "forget to protect" bug = impossible by default.
2Org header = required for all org-scoped routes
Organization: <org-uuid> header missing → 400 Bad Request. Frontend-ი Zustand store-ში ინახავს activeOrgId-ს და ყოველ request-ზე header-ში გზავნის. normalizeOrgHeader() — array reject, whitespace trim.
3Cross-tenant context = same guard, different resolution
User Org B-ის member-ია, Org A-ს resource-ს ითხოვს. Guard Layer 3: member? no → active grant? yes → pass + scope injection. DB query Layer 5: .where({ id, organizationId: 'org_a' }) — same query shape. Transparency: cross-tenant request-ი audit log-ში ჩაიწერება via_grant_id-ით.
4Role hierarchy: org_admin ⊃ billing_admin ⊃ member ⊃ viewer
org_admin includes all lower permissions. billing_admin — billing only, cannot manage members. member — read+write devices, alerts, reports. viewer — read only. Guard compares ordinal values: userRoleOrdinal >= requiredRoleOrdinal.
5System admin bypasses org guard entirely
user.role = system_admin → Layer 3 passes without membership check. But system admin still sends Organization header for context — otherwise audit log has no org attribution. System admin actions tagged actor_type: platform_operator in audit_log.
6@RequireScope() non-empty — CI enforced
Jest meta-test scans src/**/*.controller.ts: every @Get/@Post/@Patch/@Delete method must have @RequireScope([...]) with non-empty array. Empty scope → CI fail. Ceiling: 0 violations. No exception without written CTO sign-off.
7Manufacturer portal — dedicated guard, not role
Manufacturer routes use ManufacturerGuard — checks org.org_type === 'manufacturer' + membership. Regular client org member cannot access /manufacturer/* even if URL is known. Separate guard = explicit, not a permission level check.
8404 for all tenancy failures, never 403
Resource belongs to different org? → 404. Cross-tenant request without grant? → 404. Grant revoked? → 404. Exception: scope insufficient on existing cross-tenant grant → 403 (user knows they have partial access, scope mismatch is not a secret). Documented in meta-test assertions.
9Suspended org → read-only access
org.status = suspended → OrgStatusInterceptor injects readOnly: true into request context. All mutation endpoints (POST/PATCH/DELETE) return 402 Payment Required with { code: 'ORG_SUSPENDED', paymentUrl: '...' }. GET endpoints pass normally — data must remain accessible.
10Rate limit headers on every response
Every authenticated response: X-RateLimit-Limit: 2000, X-RateLimit-Remaining: 1857, X-RateLimit-Reset: 1714823460. Frontend reads these and shows quota warning in UI when Remaining < 10%. On 429: show retry countdown timer.
11Frontend route guard — Next.js middleware
Middleware runs at edge (Vercel). Checks: sessionCookie present → decode → user.status → activeOrgId. Unauthenticated → redirect /login?returnUrl=.... No org selected → redirect /select-org. System admin accessing non-admin route → allowed (no block). Middleware does NOT hit DB — cookie-only, fast.
12Deep link preservation — returnUrl
User visits /devices/abc123 without session → redirected to /login?returnUrl=%2Fdevices%2Fabc123. After successful login → router.push(returnUrl). Sanitized: returnUrl must start with /, no external domains, no javascript:. Prevents open redirect attacks.

06Redirect Map — სად გადადის

  Trigger სიტუაცია                        →  Destination
  ────────────────────────────────────────────────────────────────
  Login success, 1 org membership        →  /dashboard        (org auto-selected)
  Login success, 2+ org memberships     →  /select-org
  Login success, 0 org memberships      →  /onboarding       (create first org)
  Login success, system_admin            →  /admin
  Login success, manufacturer only       →  /manufacturer

  Visit protected route, no session      →  /login?returnUrl=...
  Visit protected route, no org          →  /select-org
  Visit /login when already logged in    →  /dashboard        (or default org)
  Visit /register when logged in         →  /dashboard

  org.status = suspended                 →  /settings/billing  (mutation blocked)
  org.status = terminated                →  /account-terminated
  user.status = pending_verification     →  /verify-email

  Token expired (JWT 401)                →  /login?session=expired
  Firebase refresh success               →  current page reload (silent)

  DELETE /me/erasure-request cancel      →  /profile
  Erasure complete (webhook)             →  logout + /erasure-confirmed

  Invite accept success (logged in)      →  /settings/grants
  Invite accept success (not logged in)  →  /register?inviteCode=...
  Invite accept — wrong org user         →  /invite/error?reason=wrong_org
  Invite expired / invalid               →  /invite/error?reason=expired

  POST /organizations success            →  /onboarding/devices  (setup wizard step 2)
  Org approval pending                   →  /onboarding/pending
  Org approved (webhook)                 →  /dashboard            (email + redirect)

  404 from API                           →  show inline error     (no full redirect)
  500 from API                           →  show error banner     (no full redirect)
  Network offline                        →  offline banner        (no redirect)
