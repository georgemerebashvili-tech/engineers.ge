02 Auto-Diagnostics — თვით-დიაგნოსტიკა

სიტემა თვითონ ადევნებს თვალყურს საკუთარ ჯანმრთელობას. ამას ეძახიან health check — სამ სიღრმეზე:

🟢Shallow
GET /health

Process alive? — 200 OK.
Load balancer / uptime monitor-ისთვის. <5ms response.
🔵Deep
GET /health/deep

DB ping · Redis ping · S3 list · MQTT broker reachable. Per-component status.
🧬Diagnostic
GET /health/diagnostic

Full system self-test: data integrity · migration status · queue depth · auth provider.
Deep health response shape
  GET /health/deep

  {
    status:    "ok" | "degraded" | "down",
    timestamp: "2026-04-23T10:00:00Z",
    version:   "1.2.3",
    uptime:    3600,
    checks: {
      postgres:  { status: "ok",       latencyMs: 4  },
      redis:     { status: "ok",       latencyMs: 1  },
      s3:        { status: "degraded", latencyMs: 850, note: "high latency" },
      mqtt:      { status: "ok",       latencyMs: 12 },
      firebase:  { status: "ok",       latencyMs: 88 }
    }
  }

  HTTP 200 → ok | HTTP 207 → degraded | HTTP 503 → down

Diagnostic-level checks

1
Migration status — typeorm migration:show equivalent — runs pending migrations count. If >0 → warning.
2
Orphan tenancy check — spot-check 10 random org_access_grants: grantor + grantee orgs exist and are active. Broken FK = data integrity issue.
3
Queue depth — SQS/BullMQ queue depth. If >1000 messages stale >5min → degraded.
4
Expired invites cleanup — count expired pending invites. If >500 → triggers background job. Diagnostic-ი არ ბლოკავს — ატყობინებს.
5
Disk / memory — container memory usage vs limit. >85% → warning. >95% → critical.
🩺
Auto-diagnostic runs: load balancer-ი shallow-ს 30s-ში ერთხელ. CloudWatch-ი deep-ს 1min-ში ერთხელ. Diagnostic-ი — deploy-ის შემდეგ ავტო, შემდეგ კვირაში ერთხელ cron.
PG-002/health/deep auth — internal tokenLocked
/health — public (no auth).
/health/deep — X-Internal-Token header, value from env HEALTH_SECRET. Load balancer passes it; public internet-ი ვერ გამოიყენებს.
/health/diagnostic — system_admin JWT or internal token.

03 Personal User Settings — პირადი ნასტროიკა

ყველა user-ს აქვს პირადი პარამეტრები, რომლებიც org-ისგან დამოუკიდებელია. ეს ნიშნავს: გიორგი TBC-ში და Silknet-ში განსხვავებული UI preference-ებით შეიძლება მუშაობდეს.

Global preferences (user-level)

  • timezone — IANA timezone string (Asia/Tbilisi)
  • locale — ka-GE, en-US, de-DE…
  • theme — light | dark | system
  • date_format — DD/MM/YYYY vs MM/DD/YYYY
  • notification_channel — email / push / both / none
  • default_org_id — სად იხსნება session login-ზე

Per-org preferences (context-level)

  • dashboard_layout — JSON blob (widget positions)
  • table_columns — per-table visible columns + order
  • alert_filter — default severity filter
  • sidebar_pinned — pinned menu items
  • report_format — PDF vs XLSX default

API endpoint
  GET   /me/preferences               → global + active org prefs merged

  On login: preferences fetched once → cached in Zustand store.

04 GDPR Right-to-Erasure — მომხმარებლის ამოშლა

⚖
სამართლებრივი მოთხოვნა. GDPR Art. 17 + საქართველოს „პერსონალური მონაცემების დაცვის შესახებ" კანონი (2024) — მომხმარებელს უფლება აქვს ითხოვოს მისი ყველა მონაცემის სრული წაშლა. 30 კალენდარული დღე ვადა. Audit log erasure-ისა თავად უნდა დარჩეს (GDPR Art. 30 — accountability).

ეს რამდენი საფეხურის გავლის process-ია:

1
მოთხოვნა — user-ი ითხოვს POST /me/erasure-request — user-ი authenticated ითხოვს. სისტემა გზავნის email confirmation-ს: "ნამდვილად გინდა? 7 დღე გაქვს გაუქმება."
2
7-დღიანი cooldown — გაუქმება შესაძლებელია Status: pending_confirmation. Cron-ი ყოველ დღე reminder email-ს გზავნის. DELETE /me/erasure-request — cancel.
3
Org admin check თუ user org_admin-ია — erasure blocked სანამ სხვა admin-ს არ გადაუქცევს. Error: "გთხოვ სხვა admin-ი დანიშნე ჯერ."
4
Billing check Open invoices? Unpaid subscription? → Erasure deferred. Email: "გთხოვ ანგარიშსწორება."
5
Erasure execution — irreversible 7 დღის cooldown + checks passed → background job-ი იწყებს.
6
Confirmation email + erasure certificate "შენი მონაცემები წაიშალა. სერტიფიკატი: ERS-2026-XXXXX." GDPR სტანდარტია.

05 Observability & Traceability — ყველაფრის ტრეკინგი

"სად რა როგორ რატომ კეთდება და რა რასთანაა დაკავშირებული" — ეს observability-ს სამი სვეტია: Traces (request path), Metrics (numbers), Logs (events). გაერთიანებული: OpenTelemetry → CloudWatch.

🔬Traces
ყოველი HTTP request-ი → distributed trace. X-Request-ID propagation. NestJS → DB → S3 → Firebase — ყველა span-ი linked. AWS X-Ray.
📊Metrics
Request count, latency p50/p99, error rate, queue depth, DB connection pool, tenant activity count. CloudWatch custom metrics.
📋Logs
Structured JSON — {timestamp, requestId, orgId, userId, action, durationMs}. PII-free (§SECURITY-004). CloudWatch Logs Insights.
