03Org Membership — წევრობა და როლი

Role hierarchy: org_admin ⊃ billing_admin ⊃ member ⊃ viewer. Roles-ი additive არ არის — ყოველი user-ი ერთ org-ში ერთ role-ს ატარებს.

org_admin ordinal: 4
  • ყველა ქვემოთ მოყვანილი ნებართვა
  • Member-ების მოწვევა / ამოშლა / role-ის შეცვლა
  • Cross-org invite გენერაცია / revoke
  • Device settings & configuration
  • Org profile update
  • Integration tokens შექმნა
  • Org-level audit log ნახვა
  • ვერ: Billing management (billing_admin-ს)
billing_admin ordinal: 3
  • Invoice ნახვა / download
  • Payment method განახლება
  • Plan upgrade / downgrade
  • Billing portal წვდომა
  • ვერ: Member management, device settings, invites
  • შენიშვნა: org_admin ასევე ახდენს billing action-ებს
member ordinal: 2
  • Device list / detail / claim
  • Commands გაგზავნა
  • Alert read + update (resolve/assign)
  • Report შექმნა / ნახვა / export
  • Calculator გამოყენება
  • ვერ: Member management, device settings, invites, billing
viewer ordinal: 1
  • Read-only: devices, alerts, reports
  • Dashboard ნახვა
  • ვერ: ყველა mutation — commands, resolve, create report, claim device
  • გამოყენება: Auditor-ები, external stakeholders

Role transition rules

ActionWho can do itConstraint
Invite new member with roleorg_adminCannot invite with higher role than own (org_admin can invite org_admin)
Change member roleorg_adminCannot demote sole org_admin — must promote another first
Remove memberorg_adminCannot remove sole org_admin
Leave org voluntarilyany memberSole org_admin cannot leave without reassigning
Change own role❌ nobodySelf-escalation impossible — guard checks actor ≠ target for role changes

Cross-tenant grant roles

Cross-tenant grant-ს scope-ი განსაზღვრავს permissions-ს (არა role). Scope enum:

read:devices read:reports read:alerts write:commands full
Cross-tenant user-ი ვერ ახდენს member management-ს, ვერ ქმნის invites-ებს, ვერ ხედავს billing-ს — scope full-ზეც კი. Grant = resource access, not org membership.

04Device — მოწყობილობის სტატუსი

State flow
unclaimed →POST /devices/claim never_seen →first MQTT connect online
online →heartbeat timeout (5min) offline →reconnect online
online →error event received error →device reports ok online
online / offline / error →admin decommission decommissioned
სტატუსიგანმარტებაTriggerAlert generated?
unclaimed Manufacturer-ის registry-შია, კლიენტს არ ეკუთვნის Manufacturer-მა დაარეგისტრირა —
never_seen Claimed, მაგრამ ჯერ არ დაუკავშირდია POST /devices/claim — (90d-ზე warning manufacturer-ს)
online MQTT connected · heartbeat OK MQTT connect event — (first_seen: manufacturer notification)
offline heartbeat timeout (default 5min) ან clean disconnect MQTT will message ან timeout ✅ severity=medium if offline > threshold
error Device-ი online-ია მაგრამ error state-ს აბრუნებს error payload on device/+/telemetry topic ✅ severity depends on error_code
decommissioned Device ამოღებულია სერვისიდან · ისტორია ინახება PATCH /devices/:id {status:"decommissioned"} by org_admin —
ℹ
Heartbeat threshold: configurable per-device (device.heartbeatIntervalSec, default 300). MQTT broker-ი will message-ს aaიგზავნის disconnect-ზე → NestJS event handler → status update + alert generation.

07Org Invite — მოწვევის სტატუსი

State flow
pending →accept before expiry accepted
pending →grantor org_admin revokes revoked
pending →expires_at passes (cron) expired
სტატუსიToken valid?Grant created?Notes
pending✅❌ not yetToken usable · single-use
accepted❌ consumed✅ active grant createdgrantee_org_id filled · accepted_at logged
revoked❌❌ or existing grant also revokedImmediate effect via Redis invalidation
expired❌❌Accept attempt → generic 404 (not "expired" — enumeration prevention)
⚠
Brute-force protection: wrong/expired/revoked code ყველა → 404 { code: "INVITE_INVALID" }. არასოდეს "expired" vs "wrong" distinction over the wire. Rate limit: 10 attempts/IP/hour.

08Erasure Request — GDPR ამოშლის სტატუსი

State flow — სრული lifecycle
pending_confirmation →7 days pass (no cancel) queued →worker picks up in_progress →all steps complete completed
pending_confirmation →user cancels (DELETE /me/erasure-request) cancelled
სტატუსიUser session?TriggerAction
pending_confirmation ✅ still active POST /me/erasure-request Daily reminder email · 7-day countdown · cancel available
queued ✅ still active 7-day cooldown expires (cron) Checks: sole admin? open invoices? → if fail → email user to resolve
in_progress ❌ revoked Worker starts execution Firebase delete · DB PII scrub · PATs revoke · memberships remove · steps_log filled
completed ❌ All steps logged OK Certificate email (ERS-YYYY-XXXXX) · erasure_requests row retained 5yr · user.status = deleted
cancelled ✅ restored DELETE /me/erasure-request within 7 days Reminder emails stop · user continues normally
⚖
GDPR Art. 17 deadline: request-დან 30 კალენდარული დღე. Internal SLA: queued → completed ≤ 72h. steps_log-ი არის proof of compliance. erasure_requests table-ი append-only — არ იშლება.

Execution steps (steps_log)

step_1 Firebase Auth account delete
step_2 All PATs revoked (Redis + DB)
step_3 All sessions invalidated (Redis)
step_4 org_memberships removed (all orgs)
step_5 user_preferences + user_org_preferences deleted
step_6 audit_log.actor_user_id → [deleted-ERS-XXXXX]
step_7 billing_invoices: name/email → [redacted] (amounts retained 7yr)
step_8 users row: PII nulled, status=deleted
step_9 erasure certificate email sent
RETAINED erasure_requests row (5yr) · billing amounts · security audit log (1yr anonymized)
