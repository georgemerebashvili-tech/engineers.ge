SLO targets: LCP < 2.5s · INP < 200ms · CLS < 0.1 · TTFB < 800ms. P75 threshold. Breach → PagerDuty alert.

③ Rate limits — API quota monitoring

Rate limit layers — სამი დონე
  Layer 1 — Global (AWS WAF / CloudFront)
    10 000 req/min per IP → 429 + Retry-After   DDoS protection

  Layer 2 — Org-level (Redis token bucket)
    plan-based quota (100–unlimited req/min)    billing enforcement
  Header: X-RateLimit-Limit / Remaining / Reset

  Layer 3 — Endpoint-specific
    POST /auth/login  → 10/min per IP         brute force
    POST /invites/accept → 10/hr per IP        token guessing
    POST /erasure-request → 3/day per user     abuse prevention

Operator dashboard-ზე: live rate-limit hit graph (429 rate per minute), top-10 orgs by quota usage, throttled IPs list.

④ CSP violations — Content Security Policy

Content-Security-Policy-Report-Only → report-uri /api/csp-report. Violations persist-ავს DB-ში (30-day retention). Dashboard: violations by directive, by page, by user-agent.

Rollout strategy: Report-Only mode პირველ 2 კვირა → violations-ის ანალიზი → enforce mode. Inline scripts პროჰიბირებულია — nonce-ბაზირებული approach Next.js middleware-ში.

02 Manufacturer Portal — მწარმოებლის პორტალი

ქარხანა/მწარმოებელი ყიდის IoT მოწყობილობებს სხვადასხვა კლიენტებზე. მას სჭირდება საკუთარი მოწყობილობების fleet-ის ხილვა — სად არიან, ჩართულია თუ არა, firmware-ის ვერსია, error rate. კლიენტის ბიზნეს-მონაცემები კი — არ ეკუთვნის.

⚠
Privacy boundary: მწარმოებელი ხედავს device-ის ოპერაციულ მეტრიკებს (online/offline, firmware, error count, uptime), მაგრამ ვერ ხედავს კლიენტ ორგანიზაციის სახელს, business data-ს, ან user-ების ინფორმაციას. Tenant isolation პრინციპი (ADR-MT-001) ვრცელდება.

Org type: manufacturer

🏭Manufacturer org
ახალი org_type: manufacturer. Self-registers ან platform invite-ით. Registration-ზე ავტომატური KYB (Know Your Business) check — საგადასახადო ID validation. Verified badge.
📋Device registry
მწარმოებელი ქმნის device model-ებს: სახელი, ფოტო, specs, firmware changelog, certifications (UL/IEC). ყოველ model-ს აქვს serial prefix pattern (DMT-SENS-*).

Device claim / registration flow

1
მწარმოებელი ატვირთავს serial range-ებს POST /manufacturer/devices/batch-register → CSV ან JSON. Fields: serial, model_id, production_date, firmware_version.
2
კლიენტი "claim" ავს მოწყობილობას POST /devices/claim → serial number. სისტემა ამოწმებს manufacturer registry-ში. Claim → device-ი ეკუთვნება კლიენტის org-ს, მაგრამ manufacturer_id link ინახება.
3
Device online event → manufacturer-ი ატყობინება Rule engine: "when device first_online → notify manufacturer". Notification: Webhook / email / dashboard badge.
4
Manufacturer ხედავს aggregate fleet stats Anonymized: "342 devices online across 47 deployments". კლიენტი ვინ არის — invisible.

Notification rules — მწარმოებლის შეტყობინებები

EE-001Manufacturer notification rule engineNew
UI-ში მწარმოებელი ქმნის წესებს (no-code rule builder):
  • WHEN device_status changes TO online FOR FIRST TIME → email + webhook
  • WHEN error_count > 10 IN 24h FOR model="SENS-T100" → alert
  • WHEN firmware_version = "v2.2.0" AND last_seen > 30d → remind update
  • WHEN device unclaimed > 90d → sales follow-up trigger
Rule engine: EventBridge Rules → Lambda evaluator → SES/SNS/Webhook. Per-manufacturer rule limit: 20.

03 Tax & Financial System — გადახდები · ინვოისი · RS.GE

პლატფორმას სჭირდება ავტომატური billing pipeline — Stripe-ის საერთაშორისო ბარათებით გადახდა, ქართული RS.GE (revenue.ge) ელ-ინვოისი, საგადასახადო ანგარიში, ნაღდი გადახდის flow.

Payment stack

💳Stripe
International cards, SEPA, Google/Apple Pay. Subscription management (monthly/annual). Usage-based billing (API calls, device count). Webhook → billing events.
🇬🇪TBC Pay / BOG Pay
Georgian local payment gateway-ები. GEL ანგარიშსწორება. Installment plans. ქართული კლიენტებისთვის preferred option.
🏦Bank Transfer
Enterprise-ებისთვის: საგადახდო ბარათი → ანგარიშის ნომერი + reference. Manual reconciliation operator dashboard-ში.

Billing pipeline — ავტომატური ციკლი

1
Usage metering — real-time ყოველ API call-ზე, device connect-ზე, storage write-ზე → usage event → SQS. Aggregator Lambda: კვირის განმავლობაში ჯამდება per-org.
2
Invoice generation — billing cycle-ის ბოლო POST /admin/billing/generate-invoices cron (1-ე რიცხვი, 00:00). Per-org: plan fee + overage. Invoice object Stripe-ში + local DB. PDF auto-generated (pdfmake).
3
RS.GE ელ-ინვოისი (Georgian e-invoice) Georgian law: B2B transactions >GEL 1000 require e-invoice via RS.GE API. POST https://api.rs.ge/rs/invoices → invoice ID. Response stored in invoices.rs_invoice_id.
4
Payment attempt Stripe auto-charges saved card. Failure → retry (D+3, D+7). BOG/TBC: payment link გაიგზავნება email-ზე. Bank transfer: 7 სამუშაო დღის ვადა.
5
Reconciliation + confirmation Payment success → invoice.status = paid → confirmation email + PDF receipt → audit log entry.
!
Overdue → suspension D+14 unpaid → org.status = suspended (read-only access). D+30 → terminated. Data retention 90d (GDPR grace).

RS.GE integration architecture

Georgian e-invoice flow
  1. Invoice generated locally
     invoices { id, org_id, amount_gel, period, line_items[] }

  2. RS.GE API call (if org.country_code = 'GE' AND amount >= 1000 GEL)
     POST https://api.rs.ge/rs/invoices
     {
       seller_tin:   "405123456",      // Sazeo-ს საგადასახადო ID
       buyer_tin:    "406789012",      // კლიენტის საგადასახადო ID
       invoice_date: "2026-04-01",
       currency:     "GEL",
       total:        2400.00,
       items: [
         { name: "Sazeo Professional — April 2026", qty: 1, price: 2400, vat_pct: 18 }
       ]
     }
     ← { rs_invoice_id: "RS-2026-00184521", status: "issued" }

  3. Local record updated
     invoices.rs_invoice_id = "RS-2026-00184521"
     invoices.rs_submitted_at = NOW()

  4. RS.GE confirmation stored — buyer sees it on revenue.ge portal

EE-003Tax compliance per jurisdictionOngoing
Georgia: 18% VAT (დღგ) B2B on digital services. RS.GE e-invoice mandatory ≥ GEL 1000.
EU: MOSS/OSS scheme — VAT of buyer's country. Stripe Tax-ი handles calculation.
US: No federal digital services tax. State-level varies (NY, TX, etc.) — Stripe Tax.
Rule: თუ org.country_code-ი ცნობილი jurisdiction-ია — auto-calculate tax. Unknown → 0% + warning to org_admin to confirm tax status.
✓
PCI DSS: Sazeo არ ინახავს card data-ს. Stripe-ი handles tokenization. Payment form: Stripe Elements (iframe) — no card number touches Sazeo servers. PCI SAQ-A compliance.
